# Technical Document ML Service

## Тема проекта

**Сервис извлечения и проверки данных из технической документации**

Пользователь загружает технические документы (PDF или изображения), ML-сервис извлекает из них структурированные данные, проводит валидацию входных данных и возвращает результат обработки. 

---

## Что реализовано на текущем этапе

### Задание №1 — объектная модель ML-сервиса

В проекте спроектирована доменная объектная модель, отражающая базовую бизнес-логику сервиса

Реализованы основные сущности предметной области:

- `User` — пользователь сервиса;
- `MLModel` — базовая ML-модель;
- `TechnicalDocumentExtractionModel` — ML-модель для обработки технической документации;
- `MLTask` — базовая ML-задача;
- `DocumentExtractionTask` — задача на обработку документов;
- `PredictionResult` — результат работы модели;
- `MLRequestHistoryRecord` — история запросов;
- `Transaction` — базовая транзакция;
- `CreditTransaction` — пополнение баланса;
- `DebitTransaction` — списание средств;
- `UploadedDocument` — загруженный документ;
- `ValidationIssue` — ошибка валидации входных данных.

### Задание №2 — структура проекта и Docker Compose

Проект приведен к воспроизводимой инфраструктурной структуре и подготовлен к запуску через Docker Compose

Описаны 4 сервиса:

- `app` — backend-приложение;
- `web-proxy` — reverse proxy на Nginx;
- `rabbitmq` — брокер сообщений;
- `database` — база данных PostgreSQL

---

### Задание №3 — хранение данных через ORM

К объектной модели подключена реальная база данных PostgreSQL с использованием ORM `SQLAlchemy`

#### Реализовано

**1 Подключение ORM и конфигурация БД**
- настроено подключение к PostgreSQL через env-переменные
- добавлены `SQLAlchemy`, `psycopg`, `python-dotenv`
- реализованы `engine`, `SessionLocal`, базовый `Base`

**2 ORM-модели и схема БД**
Описаны таблицы для ключевых сущностей предметной области:

- `users`
- `ml_models`
- `ml_tasks`
- `transactions`
- `uploaded_documents`
- `prediction_results`
- `ml_request_history`
- `task_documents`

Настроены:
- первичные ключи;
- внешние ключи;
- индексы;
- ограничения целостности;
- связи между сущностями

**3 Бизнес-операции поверх ORM**
Реализован прикладной слой сервисов:

- `user_service.py`
  - создание пользователя;
  - загрузка пользователя по `id`;
  - загрузка пользователя по `email`

- `billing_service.py`
  - пополнение баланса;
  - списание средств;
  - сохранение транзакций;
  - получение истории транзакций пользователя

- `history_service.py`
  - создание записи истории ML-запроса из задачи;
  - получение истории предиктов пользователя с сортировкой по дате

**4 Инициализация базы данных**
Реализован `init_db.py`, который:
- создает таблицы;
- добавляет демо-пользователя;
- добавляет демо-администратора;
- задает стартовый баланс;
- создает базовые ML-модели;
- выполняется автоматически при старте приложения;
- является идемпотентным

**5 Тестирование**
Добавлены тесты на основные сценарии:
- создание пользователя;
- загрузка пользователя из БД;
- пополнение баланса;
- списание баланса;
- ошибка при недостатке средств;
- сохранение истории транзакций;
- получение истории транзакций в обратном хронологическом порядке;
- создание истории ML-запросов;
- получение истории предиктов пользователя;
- идемпотентность `init_db`

## Запуск проекта
Сборка и запуск сервисов:
`docker compose up -d --build`

Проверка работоспособности приложения:
`curl http://localhost/health`

Ожидаемый результат:
`{"status":"ok"}`

## Запуск тестов
Тесты запускаются внутри контейнера `app`:
`docker compose run --rm app pytest -q`

Ожидаемый результат на данном этапе:
`14 passed`
